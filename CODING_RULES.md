# Coding Rules

## Non-negotiable

- Do not assume when the correct implementation, style target, or intended element is not explicitly known.
- If something is unclear, ask the user before making the change.
- Match existing systems first. Do not invent new visual logic, spacing logic, color logic, or component behavior without confirmation.
- For visual edits, verify the exact related pattern in the current file before changing anything.
- Be attentive and precise. Read the exact element, context, and request carefully before editing.
- Think through the request before editing: identify the intended device/context, existing breakpoints, and whether the change should apply to desktop, tablet, mobile, or all sizes.
- Fix the actual root cause, not a cosmetic symptom.
- Do not say something is fixed unless the change directly addresses the real cause of the problem.
- When the same bug pattern appears in another place, apply the same proven fix pattern there instead of improvising a new one.
- Make one tightly scoped change at a time. Do not bundle extra ideas or side changes.
- DO NOT MAKE THINGS UP.
- DO NOT IMPROVISE NEW VISUALS, PATTERNS, OR LOGIC WHEN AN EXISTING SYSTEM OR REFERENCE ALREADY EXISTS.
- IF THE EXACT IMPLEMENTATION IS NOT KNOWN, ASK FIRST.

## Practical Rule

If I do not know exactly what to change or what standard to follow, I must ask instead of guessing.
